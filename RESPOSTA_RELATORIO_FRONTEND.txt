3.1 — ENDPOINT DE LOGIN (INCONSISTÊNCIA INTERNA)
────────────────────

CONFIRMADO: O único endpoint de autenticação para o Painel Administrativo é:

  POST /api/auth/admin/login

  Body esperado:
    telefone → string, número de telefone do utilizador (ex: +244923000001)
    senha    → string, palavra-passe

  Resposta de sucesso inclui:
    token     → string JWT para usar no header Authorization: Bearer {token}
    tipo      → "Bearer"
    usuario   → objecto com dados do utilizador autenticado

O endpoint POST /api/auth/jwt/login é uma via alternativa de autenticação
técnica (utilizador/palavra-passe no formato username/password). NÃO deve
ser utilizado pelo Painel Administrativo.

Acção necessária: remover ou desactivar /auth/jwt/login de authService.js.
  O auth store que já usa /auth/admin/login está correcto.

3.6 — ENDPOINTS DE UTILIZADORES NÃO CONFIRMADOS
────────────────────

TODOS CONFIRMADOS E IMPLEMENTADOS. O conjunto completo de endpoints para a
gestão de utilizadores está agora disponível:

  GET    /api/usuarios                      listar com paginação
  GET    /api/usuarios/{id}                 detalhes do utilizador
  POST   /api/usuarios                      criar novo utilizador
  PUT    /api/usuarios/{id}                 actualizar dados do utilizador
  DELETE /api/usuarios/{id}                 desactivar utilizador (soft delete)
  PATCH  /api/usuarios/{id}/ativar          reactivar utilizador
  PATCH  /api/usuarios/{id}/desativar       desactivar utilizador
  PATCH  /api/usuarios/{id}/senha           alterar senha (sem verificar a antiga)
  GET    /api/usuarios/permissoes           listar roles disponíveis com descrições
  GET    /api/usuarios/{id}/logs            histórico de acções (funcionalidade em expansão)
  POST   /api/usuarios/reset-senha          reset simplificado (ver nota abaixo)
  POST   /api/usuarios/reset-senha/confirmar retorna 501 (fluxo de email não configurado)

Notas sobre endpoints de reset de senha:
  O endpoint POST /usuarios/reset-senha aceita o pedido mas não envia email,
  pois o servidor SMTP não está configurado neste ambiente. Para redefinição
  administrativa de senha, usar PATCH /usuarios/{id}/senha que não exige
  verificação da senha antiga.
  O endpoint POST /usuarios/reset-senha/confirmar retorna erro HTTP 501
  (Not Implemented) com mensagem explicativa. O frontend deve tratar este
  retorno e apresentar ao utilizador a instrução de contactar o administrador.

PARTE 5 — PAYLOADS DE REFERÊNCIA PARA INTEGRAÇÃO
────────────────────────────────────────────────────────────────────────────────

Estruturas de payload para os pontos que geraram dúvidas ou correcções.

────────────────────
POST /api/auth/admin/login
────────────────────
  Body:
    telefone → string, obrigatório
    senha    → string, obrigatório

  Resposta (data):
    token       → string JWT
    tipo        → "Bearer"
    usuario     → objecto com id, username, nomeCompleto, telefone, roles

  Utilização do token em chamadas subsequentes:
    Header: Authorization com valor "Bearer " seguido do token

────────────────────
POST /api/usuarios (criar utilizador)
────────────────────
  Body:
    username     → string, obrigatório, entre 3 e 50 caracteres, único no sistema
    senha        → string, obrigatório, mínimo 6 caracteres
    email        → string, formato email, opcional mas recomendado
    nomeCompleto → string, opcional
    telefone     → string, opcional
    roles        → array de strings, obrigatório, mínimo um elemento

  Valores válidos para roles:
    Consultar GET /api/usuarios/permissoes para a lista completa e actualizada.
    Exemplo de resposta: { "ADMIN": "Administrador", "GERENTE": "Gerente", ... }

────────────────────
PUT /api/usuarios/{id} (actualizar utilizador)
────────────────────
  Body (todos os campos são opcionais — enviar apenas o que se pretende alterar):
    email        → string, formato email
    nomeCompleto → string
    telefone     → string
    roles        → array de strings (se omitido, os roles não são alterados)

────────────────────
PATCH /api/usuarios/{id}/senha (alterar senha — administrativo)
────────────────────
  Body:
    novaSenha → string, obrigatório, mínimo 6 caracteres

  Nota: este endpoint não exige a senha actual. Destina-se a reset administrativo.

────────────────────
POST /api/fundos/{token}/recarregar
────────────────────
  Body:
    valor       → decimal, obrigatório, valor a adicionar (deve ser positivo)
    observacoes → string, opcional, nota descritiva sobre a recarga

  Nota: o campo chama-se observacoes (não descricao).

────────────────────
GET /api/usuarios/permissoes
────────────────────
  Sem body nem parâmetros.

  Resposta (data): objecto chave-valor onde a chave é o código da role e o
    valor é a descrição em português.
    Exemplo: { "ADMIN": "Administrador do Sistema", "GERENTE": "Gerente", ... }

────────────────────
GET /api/usuarios/{id}/logs
────────────────────
  Sem body. Parâmetros opcionais de paginação via query string.

  Resposta (data): lista de eventos (actualmente retorna lista vazia).
    A funcionalidade de auditoria por utilizador está em expansão.
    Tratar lista vazia como estado válido, não como erro.
